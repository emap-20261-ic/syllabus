### 1.1 Tipos auxiliares: `enum`

Além do `struct item` já visto, o programa define dois `enum`:

```c
typedef enum {
  CMD_PRINT_MENU,
  CMD_TOTAL,
  CMD_UNKNOWN
} command;
```

`enum` cria constantes inteiras com nomes legíveis. `CMD_PRINT_MENU == 0`,
`CMD_TOTAL == 1`, etc. O compilador atribui os valores automaticamente a partir
de zero. Usar nomes em vez de números mágicos torna o `switch` em `main`
autoexplicativo.

```c
typedef enum {
  EXIT_OK          = 0,
  EXIT_OPEN        = 1,
  EXIT_ITEM_COUNT  = 2,
  EXIT_TOO_MANY    = 3,
  EXIT_ITEM_FORMAT = 4,
  EXIT_UNKNOWN_CMD = 5
} exit_code;
```

Aqui os valores são explícitos porque a convenção do sistema operacional é
`0 = sucesso`. Exportar esses códigos como `enum` em vez de `#define` evita
conflitos de nome e aparece nos erros do compilador com o nome legível.

### 1.2 Protótipos de funções

```c
void    to_lower_case(string s);
item*   find_item(item items[], int size, string name);
int     read_menu(FILE *fp, item items[], int max_items);
float   read_order(FILE *fp, item items[], int size, int *found);
command parse_command(string s);
```

C compila de cima para baixo. Sem esses protótipos, ao encontrar a chamada
`read_menu(...)` em `main`, o compilador ainda não teria visto a definição da
função e emitiria um erro. Os protótipos são declarações antecipadas: descrevem
assinatura sem dar o corpo.

### 1.3 Main

Na `main` temos a validação das entradas, leitura dos arquivos e
orquestração das chamadas para as funções auxiliares.

Três responsabilidades bem separadas: (1) validar argumentos, (2) carregar o
menu do arquivo, (3) despachar o comando. O `switch` sobre o `enum` `command`
garante que todos os casos sejam tratados — se adicionarmos um novo comando ao
`enum` e esquecermos de tratá-lo no `switch`, o compilador (com `-Wall`) avisa.

Usamos `to_lower_case` para modificar a string *in place*. Como arrays
decaem para ponteiro quando passados a funções, `s` é um ponteiro para
o primeiro char do array original — as alterações afetam o
chamador. `tolower` (de `<ctype.h>`) converte um único `char`; o laço
aplica isso a todos os caracteres.

### 1.4 Pontos para detalhar

**`fgets` vs `scanf`**: `fgets(line, N, fp)` lê no máximo `N-1` caracteres e
para na primeira quebra de linha, guardando tudo em `line`. Isso protege contra
buffer overflow e mantém o controle de onde estamos no arquivo.

**O especificador `%49[^:]`**: dentro de `sscanf`, `[^:]` é um "scanset" — lê
caracteres que *não* sejam `:`. O `49` limita o tamanho para não ultrapassar o
buffer `name[MAX_NAME_SIZE]`. O formato completo `"%49[^:]:%f"` lê: nome
(tudo antes de `:`), o próprio `:`, e depois o preço como float. `sscanf`
retorna o número de campos preenchidos com sucesso; checar se é `2` valida o
formato.

A `find_item` retorna `item*` em vez de `item`. Isto tem duas
vantagens: (1) `NULL` sinaliza "não encontrado" sem precisar de um
parâmetro extra; (2) o ponteiro aponta para o elemento original no
array, evitando cópia.

A função `read_order` precisa devolver dois valores: o total (float) e
a contagem de itens encontrados. Em C, uma função tem um único valor
de retorno; o segundo sai via ponteiro (`int *found`). O chamador
passa `&found` e a função escreve em `*found`.

`it->price` é açúcar para `(*it).price`: como `it` é ponteiro para
`item`, primeiro dereferenciamos e depois acessamos o campo. O
operador `->` combina os dois passos.

`strcspn(line, "\n")` retorna o índice do primeiro `\n` na string (ou
`strlen(line)` se não houver). Atribuir `'\0'` nessa posição remove a
quebra de linha que `fgets` inclui.

## 2. Recursão

Uma função é **recursiva** quando chama a si mesma. Toda solução recursiva tem
duas partes obrigatórias:

- **Caso base**: a condição que interrompe a recursão e retorna diretamente.
- **Caso recursivo**: chama a função com uma instância *menor* do problema.

### 2.1 Fatorial

A definição matemática já é recursiva:

$$0! = 1 \qquad n! = n \times (n-1)!$$

Em C:

```c
int fatorial(int n) {
  if (n == 0) return 1;        // caso base
  return n * fatorial(n - 1);  // caso recursivo
}
```

Trace de `fatorial(4)`:

```
fatorial(4)
  = 4 * fatorial(3)
  = 4 * 3 * fatorial(2)
  = 4 * 3 * 2 * fatorial(1)
  = 4 * 3 * 2 * 1 * fatorial(0)
  = 4 * 3 * 2 * 1 * 1  = 6
```

Cada chamada *aguarda* o retorno da chamada interna antes de continuar.

### 2.2 A pilha de chamadas

Cada chamada de função cria um **frame** na pilha: um bloco de memória com as
variáveis locais e o endereço de retorno. Para `fatorial(4)`, cinco frames são
criados em sequência; depois do `fatorial(0)` retornar `1`, os frames são
desempilhados um a um.

Se o caso base nunca for atingido, os frames acumulam indefinidamente até
esgotar a memória da pilha: **stack overflow**.

### 2.3 Como pensar recursivamente

Não tente rastrear toda a árvore de chamadas mentalmente. Em vez disso, use o
seguinte raciocínio:

> Assuma que `fatorial(n-1)` já devolve `(n-1)!` corretamente. Como chegar
> a `n!`? Multiplicando por `n`.

Esse raciocínio — assumir que a chamada recursiva funciona e usá-la para
construir a solução do problema maior — é análogo à indução matemática.


## 3. Selection sort

O **selection sort** (ordenação por seleção) é um algoritmo $O(n^2)$
simples e intuitivo.

Em cada passo, encontramos o menor elemento da parte ainda não ordenada e o
colocamos na posição correta:

```
passo 0: encontra o mínimo em a[0..n-1], troca com a[0]
passo 1: encontra o mínimo em a[1..n-1], troca com a[1]
...
passo i: encontra o mínimo em a[i..n-1], troca com a[i]
```

Após o passo `i`, os elementos `a[0..i]` estão na posição final.

Na implementação em C, o laço externo percorre as posições de destino
(`i`). O laço interno encontra o índice do mínimo no segmento
`a[i..n-1]`. Em seguida, trocamos `a[i]` com `a[min_idx]` usando uma
variável temporária `tmp` — necessária porque sem ela o valor original
de `a[i]` seria sobrescrito antes de ser copiado.

No passo `i` o laço interno executa `n - i - 1` comparações. O total
é:

$$T(n) = (n-1) + (n-2) + \cdots + 1 = \frac{n(n-1)}{2} = O(n^2)$$

Diferentemente da busca linear (melhor caso $O(1)$), o selection sort
executa o mesmo número de comparações para qualquer entrada. Mesmo que
o array já esteja ordenado, o algoritmo faz todos os passos.
