## 1. Por que `==` não compara strings

Intuitivamente, parece natural escrever:

```c
char s[100], t[100];
// ...
if (s == t) { ... }
```

Mas em C, `s` e `t` são ponteiros para o primeiro elemento de cada array. O
operador `==` compara os **endereços**, não o conteúdo. Como `s` e `t` são
buffers distintos na memória, `s == t` é sempre falso — independente do que
foi digitado.

O arquivo `compare1.c` demonstra isso:

```c
char *a = s;
char *b = t;

printf("%s\n",  a == b  ? "same" : "different");  // endereços: sempre "different"
printf("%s\n", *a == *b ? "same" : "different");  // primeiro char: não é suficiente
```

A segunda linha (`*a == *b`) compara apenas o **primeiro caractere** de cada
string — também insuficiente para saber se as strings são iguais.

### O caso curioso dos string literals

`compare3.c` mostra um caso que pode surpreender:

```c
char *s = "hello";
char *t = "hello";

printf("s = %p  t = %p\n", s, t);
printf("%s\n", strcmp(s,t) == 0 ? "same" : "different");
```

Dependendo do compilador e das otimizações, `s` e `t` podem ter **o mesmo
endereço**: o compilador detecta que os dois literais são idênticos e
armazena apenas uma cópia. Nesse caso, `s == t` seria verdadeiro por
coincidência — mas por razão errada. Nunca confie nisso para comparar
strings.

## 2. `strcmp`: comparação correta de strings

A função correta para comparar strings é `strcmp`, declarada em `<string.h>`.
Documentação:

```
man 3 string   # lista funções da família
man 3 strcmp   # detalhes de strcmp
```

Contrato de `strcmp(s, t)`:

| Resultado | Significado                               |
|-----------|-------------------------------------------|
| `0`       | strings iguais                            |
| `< 0`     | `s` vem antes de `t` (lexicograficamente) |
| `> 0`     | `s` vem depois de `t`                     |

`compare2.c` mostra o uso básico: lê duas strings do usuário, imprime os
endereços (para evidenciar que são diferentes) e o resultado de `strcmp`.

## 3. Implementação própria de `strcmp`

Para entender como `strcmp` funciona por dentro, `compare4.c` traz uma
implementação manual:

```c
int my_strcmp(char *a, char *b) {
  while ((*a != '\0' && *b != '\0') && *a == *b) {
    printf(".");
    a++;
    b++;
  }
  return (*a - *b);
}
```

Passo a passo:

- O laço avança enquanto os caracteres correntes são iguais **e** nenhuma
  string terminou (`'\0'`).
- Ao sair do laço, `a` e `b` apontam para o primeiro ponto de divergência (ou
  para o `'\0'` de uma das strings, se uma é prefixo da outra).
- O retorno `*a - *b` dá a diferença dos códigos ASCII nesse ponto: zero se
  iguais, negativo ou positivo conforme a ordem.

O `printf(".")` é apenas um traço de execução para visualizar quantas
iterações ocorrem — não faz parte do algoritmo real. Nos permite ver
que esta função não é executa em um passo, tem uma complexidade que
poderia ser descrita como uma função a partir do tamanho das strings
passadas.

Ao comparar `my_strcmp` com `strcmp` sobre as mesmas entradas, os resultados
coincidem. Isso ilustra que a biblioteca padrão não faz "mágica": é
aritmética de ponteiros combinada com a convenção do `'\0'` terminador.

## 4. Operador ternário `EXP ? a : b`

C oferece uma expressão condicional compacta:

```c
int r = (i == j ? 10 : 20);
```

Lê-se: "se `i == j`, o valor da expressão é `10`; caso contrário, `20`".

É equivalente a:

```c
int r;
if (i == j) r = 10;
else        r = 20;
```

A diferença é que `EXP ? a : b` é uma **expressão** — produz um valor e pode
ser usada em qualquer lugar onde um valor é esperado (inicialização, argumento
de função, etc.). O `compare1.c` usa isso no próprio `printf`:

```c
printf("%s\n", a == b ? "same" : "different");
```

## 5. Complexidade de algoritmos

Quando avaliamos um algoritmo, queremos saber duas coisas: ele está
correto? E ele é eficiente no uso de **recursos** — tempo e memória?

A pergunta prática é: como comparar dois programas que resolvem o mesmo
problema? Medir o tempo de execução na máquina é uma opção — chamamos isso
de **avaliação empírica** — mas ela depende do hardware e é difícil de
replicar. O comando `time` do shell dá uma ideia:

```
> time ./sum 10000000
  0.02s user ...
> time ./sum 1000000000
  1.25s user ...
```

Ao multiplicar a entrada por 100, o tempo cresce proporcionalmente. Isso
sugere que o custo do algoritmo cresce como uma **função do tamanho da
entrada** — e é exatamente isso que queremos caracterizar de forma
independente da máquina.

### O exemplo da lista telefônica

Três estratégias para encontrar um nome em uma lista telefônica
ordenada:

1. Percorrer página por página — para $n$ páginas, no pior caso
   verifica $n$ páginas.
2. Avançar duas páginas de cada vez — no pior caso verifica $n/2$
   páginas; melhor, mas ainda proporcional a $n$.
3. Abrir no meio, descartar metade — a cada passo o problema é
   dividido ao meio.

As estratégias 1 e 2 são muito parecidas quando $n$ é grande: ambas
são $O(n)$. A estratégia 3 é fundamentalmente diferente: $O(\log
n)$. Para uma lista de um bilhão de nomes, $\log_2(10^9) \approx 30$ —
trinta passos contra um bilhão.

### Notação O 

A formalização dessa ideia — a **Notação $O$** nos permite colocar
funções em classes de equivalência.  O essencial que já podemos
guardar: o que importa não é o tempo absoluto, mas **como o tempo
cresce** à medida que a entrada cresce. Fatores constantes (velocidade
do hardware, detalhes de implementação) são irrelevantes para essa
análise.
