## 1. Problema, algoritmo e implementação

Antes de aprofundar a Notação $O$, é importante distinguir três conceitos que
frequentemente se confundem:

- **Problema**: a especificação do que se quer resolver — a entrada e a saída
  esperada, sem dizer *como* chegar lá.
- **Algoritmo**: uma sequência finita de passos bem definidos que resolve o
  problema. Um mesmo problema pode ter vários algoritmos distintos.
- **Implementação**: a tradução de um algoritmo em código numa linguagem de
  programação. Um mesmo algoritmo pode ter várias implementações, inclusive em
  linguagens diferentes.

Algoritmos podem ser analisados sob diferentes ângulos:

- **Correção**: o algoritmo sempre produz a resposta certa?
- **Terminação**: o algoritmo sempre para?
- **Eficiência**: quanta memória consome? Quanta energia? Quanto tempo?

Nesta disciplina vamos focar em **tempo**.

## 2. Medir tempo de execução não é suficiente

A abordagem mais direta seria medir o tempo de relógio com `time`:

```
> time ./sum 10000000
  0.02s user ...
> time ./sum 1000000000
  1.25s user ...
```

O problema: esse número depende do hardware. O mesmo programa roda em tempos
completamente diferentes num smartphone e num servidor de última geração. Não
podemos comparar algoritmos dessa forma.

A solução é aproximar o custo pelo número de **instruções executadas**. Essa
contagem é independente do hardware e nos dá uma medida comparável entre
máquinas.

## 3. A função de custo T(n)

Depois de analisar os passos que um algoritmo $P$ executa, obtemos uma função:

$$T : n \to n$$

que mapeia o **tamanho da entrada** para o **número de passos** que $P$
executa.

### O que é "tamanho da entrada"?

Depende do problema:

- Para uma lista de números: pode ser a *quantidade* de elementos.
- Para um número inteiro: pode ser o *tamanho da representação*, isto é, a
  quantidade de dígitos (ou bits). Veja `sum.c`: o loop executa `n` vezes, onde
  `n` é o *valor* do número passado — mas esse valor pode ser representado em
  apenas $\log_{10}(n)$ dígitos.

Sempre deixe claro qual das duas interpretações está sendo usada.

## 4. Melhor caso e pior caso

Para o mesmo algoritmo e o mesmo tamanho de entrada $n$, o número de passos
pode variar de acordo com o *conteúdo* da entrada. Por isso falamos de:

- **Melhor caso**: a entrada mais favorável para o algoritmo.
- **Pior caso**: a entrada que força o máximo de passos.

**Exemplo**: dado um arquivo com uma lista de números (um por linha), encontre
o *primeiro* valor menor que 10, ou responda "FALHA".

```
para cada linha
  lê o valor e guarda em n
  compara n com 10
    se n >= 10: continua
    se  n < 10: para e responde ACHEI
responde FALHA
```

- **Melhor caso**: o primeiro número já é menor que 10 — o algoritmo para após
  1 passo.
- **Pior caso**: nenhum número é menor que 10 — o algoritmo percorre todos os
  $n$ elementos.

Em análise de complexidade, costumamos nos preocupar com o **pior caso**: ele
garante um limite superior sobre o tempo de execução.

## 5. Notação O

A **Notação $O$** (lê-se "O grande") formaliza a ideia de "como o tempo
cresce". Ela agrupa funções em classes de equivalência onde fatores
multiplicativos constantes são ignorados — afinal, constantes dependem do
hardware e não são relevantes para comparar algoritmos.

As classes mais comuns, em ordem crescente de custo:

| Classe        | Exemplo de $T(n)$   | Descrição       |
|---------------|---------------------|-----------------|
| $O(1)$        | $T(n) = 42$         | tempo constante |
| $O(\log n)$   | $T(n) = 3\log_2 n$  | logarítmico     |
| $O(n)$        | $T(n) = 5n + 7$     | linear          |
| $O(n \log n)$ | $T(n) = 2n\log_2 n$ | linearítmico    |
| $O(n^2)$      | $T(n) = n^2 + 100n$ | quadrático      |
| $O(2^n)$      | $T(n) = 2^n$        | exponencial     |

Para $n$ grande, a classe domina: um algoritmo $O(n^2)$ eventualmente supera
qualquer $O(n)$, independentemente das constantes.

## 6. `sizeof` e o tamanho de arrays

O operador `sizeof` retorna o tamanho em bytes de um tipo ou de uma variável.
Ele pode receber um tipo ou uma expressão:

```c
sizeof(int)        // tamanho do tipo int
sizeof(arr)        // tamanho total do array arr em bytes
sizeof(arr[0])     // tamanho de um elemento
```

O arquivo `sizeof.c` ilustra as diferenças:

```c
int arr[5] = {1, 2, 3, 4, 5};
int *p = arr;
printf("%zu %zu %zu %zu\n",
       sizeof(arr),       // 20 (5 × 4 bytes)
       sizeof(p),         // 8 (tamanho de um ponteiro, não do array!)
       sizeof(arr[0]),    // 4 (um int)
       sizeof(*(p + 1))); // 4 (um int)
```

**Atenção**: quando um array é passado para uma função, ele decai para um
ponteiro — `sizeof` dentro da função dá o tamanho do ponteiro, não do array
original.

### Calculando o número de elementos de um array

O idioma padrão em C para obter o número de elementos:

```c
int n = sizeof(numbers) / sizeof(numbers[0]);
```

Em `numbers1.c` isso aparece explicitamente:

```c
int numbers[] = {4, 6, 8, 2, 7, 5, 0, 10};
int n = sizeof(numbers) / sizeof(numbers[0]);

for (int i = 0; i < n; i++) { ... }
```

Por que preferir `sizeof(numbers[0])` a `sizeof(int)`? Porque se o tipo do
array mudar (de `int` para `long`, por exemplo), o código continua correto sem
nenhuma alteração.

### Hard-code do tamanho é um problema de manutenibilidade

Escrever o tamanho diretamente no loop:

```c
for (int i = 0; i < 8; i++) { ... }   // ruim: 8 espalhado pelo código
```

é um problema: se o array crescer, é preciso lembrar de atualizar *todos* os
lugares onde esse número aparece. Duas alternativas melhores:

```c
// 1. Variável global (uma única fonte de verdade)
int SIZE = 8;

// 2. Constante de pré-processador
#define SIZE 8
```

Com `#define`, o pré-processador substitui `SIZE` por `8` antes da compilação
— não ocupa memória como uma variável.

## 7. Busca linear — `src/numbers1.c`

A busca linear percorre o array do início ao fim até encontrar o elemento
procurado (ou esgotar todos os elementos):

```c
int numbers[] = {4, 6, 8, 2, 7, 5, 0, 10};
int n = sizeof(numbers) / sizeof(numbers[0]);

for (int i = 0; i < n; i++) {
  if (numbers[i] == t) {
    printf("Found at %d\n", i);
    return 0;
  }
}
printf("Not found\n");
```

- **Melhor caso**: elemento na posição 0 — 1 comparação, $O(1)$.
- **Pior caso**: elemento na última posição ou ausente — $n$ comparações,
  $O(n)$.

O mesmo raciocínio se aplica a `names0.c`, que faz busca linear em um array de
strings usando `strcmp`:

```c
string names[] = {"Bill", "Charlie", "Fred", "George", "Ginny", "Percy", "Ron"};

for (int i = 0; i < 7; i++) {
  if (strcmp(names[i], argv[1]) == 0) {
    printf("Found\n");
    return 0;
  }
}
printf("Not found\n");
```

## 8. Busca binária — `src/numbers2.c`

A busca binária só funciona em arrays **ordenados**. A cada passo ela compara o
elemento buscado com o elemento do meio e descarta metade do array restante:

```c
int binary_search(int a[], int n, int x) {
  int low = 0, high = n - 1;
  while (low <= high) {
    int k = (high + low) / 2;
    if (a[k] == x)
      return k;
    else if (a[k] < x)
      low  = k + 1;
    else
      high = k - 1;
  }
  return -1;  // não encontrado
}
```

Passo a passo:

1. `low` e `high` delimitam a faixa de busca atual.
2. `k` é o índice do meio.
3. Se `a[k] == x`: encontrado, retorna o índice.
4. Se `a[k] < x`: o valor está na metade direita; `low = k + 1`.
5. Se `a[k] > x`: o valor está na metade esquerda; `high = k - 1`.
6. Quando `low > high`, a faixa está vazia — retorna `-1`.

A cada iteração o espaço de busca é **dividido ao meio**. Para $n$ elementos,
o pior caso requer $\log_2 n$ iterações: $O(\log n)$.

### Comparação

| Algoritmo     | Pré-condição   | Pior caso   |
|---------------|----------------|-------------|
| Busca linear  | nenhuma        | $O(n)$      |
| Busca binária | array ordenado | $O(\log n)$ |

Para $n = 10^9$ (um bilhão), busca linear pode exigir $10^9$ comparações;
busca binária precisa de no máximo $\log_2(10^9) \approx 30$.

O custo de ordenar o array previamente é $O(n \log n)$ — mas se muitas buscas
serão feitas, vale o investimento.
