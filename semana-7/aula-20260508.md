### `scanf` e seus perigos

A função `scanf` lê de `stdin` segundo um template (`"%i"`, `"%s"`, etc).
Para cada designador no template, é preciso passar **um endereço** onde o
valor lido será gravado — daí o `&` em `scanf("%i", &x)`. Sem `&`, `scanf`
recebe o valor da variável (lixo) em vez do endereço, e grava em local
arbitrário.

A progressão `scanf0.c` → `scanf3.c` mostra uma sequência de tentativas
de ler uma **string** com `scanf` e por que isso é problemático.

#### `scanf0.c` — funcionou, mas é texto curto

```c
int x = 0, y = 0;
scanf("%i:%i", &x, &y);
printf("(%i,%i)\n", x, y);
```

Para inteiros, `scanf` é OK: o tamanho de um `int` é fixo e conhecido.

#### `scanf1.c` — bus error

```c
char *s = NULL;
scanf("%s", s);
```

`s` é `NULL` — não há buffer para receber a entrada. `scanf` tenta
escrever em endereço inválido e o programa morre com *bus error* ou
*segmentation fault*. Pergunta natural: **quanto espaço alocar?** A
resposta honesta é "não dá para saber em geral", e é exatamente esse o
problema do `scanf` com strings.

#### `scanf2.c` — alocou, mas pode estourar

```c
char *s = malloc(10);
scanf("%s", s);
```

Funciona se a entrada tiver até 9 caracteres + `\0`. Se o usuário digitar
20 caracteres, `scanf` escreve além do buffer — *buffer overflow*. Esse é
o vetor clássico de exploits em programas em C.

#### `scanf3.c` e `fgets` — limitando a leitura

```c
char s[10];
scanf("%9s", s);
```

`%9s` diz a `scanf` para ler no máximo 9 caracteres. Mais idiomático
ainda é usar `fgets`, que lê uma linha respeitando o tamanho do buffer:

```c
fgets(s, sizeof(s), stdin);
```

`fgets` é a alternativa segura a `gets` (que foi removida do C padrão por
não ter como limitar leitura). Veja `man fgets` e `man gets` para os
detalhes.

**Moral:** ao ler strings, controle o tamanho. `scanf("%s", …)` sem
limite é um bug esperando para acontecer.


### Lendo e escrevendo arquivos — `phonebook.c`

Todas as funções de I/O em arquivo ficam em `<stdio.h>`:

| função               | papel                                    |
|----------------------|------------------------------------------|
| `fopen`              | abre arquivo, retorna `FILE*`            |
| `fclose`             | fecha (sempre!)                          |
| `fgetc` / `fputc`    | lê/escreve um caractere                  |
| `fgets` / `fputs`    | lê/escreve uma linha                     |
| `fread` / `fwrite`   | lê/escreve N bytes brutos                |
| `fscanf` / `fprintf` | versões "formatadas" de `scanf`/`printf` |

`src/phonebook.c` aplica isso num programa real: uma agenda armazenada em
CSV. Pontos a destacar:

- **`fgets` com `sizeof(line)`**: o mesmo idioma seguro do `scanf3.c`.
- **`%[^,]`**: scanset que casa "tudo até a vírgula" — sintaxe
  poderosa do `sscanf`.
- **Truque do `\n` no fim do `fgets`**: `fgets` deixa o `\n` no buffer.
  A função `fixme` substitui o primeiro `\n` por `\0` usando
  `strcspn(b, "\n")`.
- **Salvar é `fprintf`**: `fprintf(file, "%s,%s\n", p.name, p.number)` —
  exatamente como `printf`, mas escrevendo no arquivo aberto em modo
  `"w"`.

### Há várias formas de "ler" um arquivo

A escolha depende do que você quer fazer:

| forma           | unidade           | quando usar                         |
|-----------------|-------------------|-------------------------------------|
| `fgetc`         | um caractere      | percorrer texto byte a byte         |
| `fgets`         | uma linha         | arquivos de texto linha a linha     |
| `fread`         | N bytes brutos    | formatos binários (imagens, áudio)  |
| `fscanf`        | campo formatado   | parsing simples de texto estruturado|

A diferença não é só ergonômica — a **interpretação** dos bytes muda. As
três primeiras retornam bytes; o `fscanf` interpreta segundo o template.
Mas mesmo `fgetc` e `fgets`, embora retornem bytes, **assumem
implicitamente texto** (o tipo de retorno é `char`/`char*`). Para arquivos
binários, é preciso pensar em bytes "crus" — é o tema da próxima seção.

### Byte ≠ char — `src/hexdump.c`

Para preparar o terreno para formatos binários, escrevemos um mini hex
dump:

```c
#include <stdint.h>
#include <stdio.h>

typedef uint8_t BYTE;

int main(int argc, char *argv[]) {
  if (argc != 2) return 1;
  FILE *f = fopen(argv[1], "r");
  if (!f) return 1;

  BYTE b;
  int n = 0;
  while (fread(&b, sizeof(BYTE), 1, f) == 1 && n < 64) {
    printf("%02x ", b);
    if (++n % 16 == 0) printf("\n");
  }
  printf("\n");
  fclose(f);
}
```

Rodando em um arquivo de texto:

```
$ ./hexdump cat.c
23 69 6e 63 6c 75 64 65 20 3c 73 74 64 69 6f 2e
68 3e 0a 0a 69 6e 74 20 6d 61 69 6e 28 69 6e 74
...
```

Os bytes são ASCII imprimível (`23`=`#`, `69`=`i`, `6e`=`n`, …).

Rodando numa imagem:

```
$ ./hexdump lecture.jpg
ff d8 ff c0 00 11 08 05 00 07 80 03 01 11 00 02
...
```

Os bytes começam em `ff d8 ff` — a *magic number* do JPEG — e seguem com
valores arbitrários, muitos fora da faixa imprimível. **O arquivo não é
texto**; tentar interpretá-lo como tal não faz sentido.

### Bug clássico — `src/cat.c` em arquivo binário

O `cat.c` tem o loop "ingênuo" de leitura caractere a caractere:

```c
char ch;
while ((ch = fgetc(ptr)) != EOF)
  printf("%c", ch);
```

Funciona perfeitamente em arquivos de texto. Mas:

```
$ ./cat lecture.jpg | wc -c
       0
```

**Zero bytes!** Por quê?

`fgetc` retorna `int` justamente para conseguir representar 256 valores
de byte (0 a 255) **mais** o sentinela `EOF`, que é tipicamente `-1`. Ao
guardar o resultado em `char ch`, com `char` sendo `signed char` por
padrão em macOS/x86 (faixa −128 a 127), bytes acima de 127 viram números
negativos:

| byte (hex)            | byte (decimal) | como `signed char`   |
|-----------------------|----------------|----------------------|
| `0x41` (`A`)          | 65             | 65                   |
| `0x7F`                | 127            | 127                  |
| `0x80`                | 128            | −128                 |
| `0xC3` (acento UTF-8) | 195            | −61                  |
| `0xFE`                | 254            | −2                   |
| `0xFF`                | 255            | −1 ← colide com EOF! |

O loop `while (ch != EOF)` só falha quando `ch == −1`, ou seja,
**exatamente no byte `0xFF`**. Bytes 128–254 também viram negativos,
mas nenhum coincide com `EOF`, então o laço continua e o `printf("%c",
ch)` re-converte para `unsigned char` na impressão e o byte original
sai intacto no terminal.

E qual é o **primeiro byte de qualquer JPEG**? `0xFF`. O loop sai antes
de imprimir um único byte.

#### Por que ASCII e até UTF-8 com acentos sobrevivem

- **ASCII puro (≤ 127):** nenhum byte chega a 0xFF → funciona.
- **UTF-8 com acentos (`á`, `ç`, `ã`, `é`…):** funciona por
  coincidência.  UTF-8 produz bytes 0x80–0xF7 nas sequências
  multi-byte, mas a especificação **proíbe 0xFE e 0xFF**. Verificamos
  no hex dump de `"olá mundo\nção\n"`:

  ```
  o  l  á        m  u  n  d  o  \n  ç     ã     o  \n
  6f 6c c3 a1 20 6d 75 6e 64 6f 0a c3 a7 c3 a3 6f 0a
  ```

  Os `0xC3`, `0xA1`, `0xA7`, `0xA3` viram números negativos no
  `signed char`, mas nenhum é −1. O `./cat` imprime tudo e o terminal,
  que sabe decodificar UTF-8, mostra `olá mundo / ção` corretamente.

- **Arquivo binário qualquer:** é só questão de tempo até aparecer um
  `0xFF`. Demonstração com `printf 'AAA\xffBBB' > /tmp/x`:

  ```
  $ ./hexdump /tmp/x
  41 41 41 ff 42 42 42

  $ ./cat /tmp/x
  AAA          ← parou no 0xFF, perdeu os 4 bytes seguintes
  ```

#### Conserto

Usar `int ch` (o tipo que `fgetc` devolve), ou usar
`unsigned char`/`uint8_t` no buffer (como faz `hexdump.c`).

**Moral**: o tipo `char` carrega a hipótese implícita "isto é texto", e
em particular "byte `0xFF` não vai aparecer". Para ler bytes brutos, use
`unsigned char` ou `uint8_t`.


### Magic numbers — `src/jpeg.c`

Cada formato de arquivo tem uma **assinatura** (*magic number*) nos
primeiros bytes que serve para identificá-lo. Alguns exemplos:

| formato | magic number                               |
|---------|--------------------------------------------|
| JPEG    | `FF D8 FF`                                 |
| PNG     | `89 50 4E 47 0D 0A 1A 0A`                  |
| GIF     | `47 49 46 38 37 61` (`GIF87a`) ou `GIF89a` |
| BMP     | `42 4D` (`BM`)                             |
| PDF     | `25 50 44 46 2D` (`%PDF-`)                 |

O comando `file` no terminal usa exatamente essa técnica para
identificar tipos de arquivo, com uma tabela enorme de assinaturas.


### Formato BMP e `__attribute__((packed))`

O exemplo `src/filter/` lê arquivos BMP. O cabeçalho é uma sequência fixa
de campos definida pela Microsoft, e está modelado como struct em
`bmp.h`:

```c
typedef struct {
  WORD   bfType;       // 2 bytes  - sempre "BM" (0x4d42)
  DWORD  bfSize;       // 4 bytes  - tamanho do arquivo
  WORD   bfReserved1;  // 2 bytes
  WORD   bfReserved2;  // 2 bytes
  DWORD  bfOffBits;    // 4 bytes  - offset dos pixels
} __attribute__((packed)) BITMAPFILEHEADER;
```

Total: **14 bytes**. O `__attribute__((packed))` é o detalhe não-óbvio
que faz a leitura funcionar.

#### Padding: o que o compilador faz por padrão

Por padrão, o compilador insere bytes de **padding** entre campos para
que cada um caia num endereço alinhado ao seu tamanho natural (`uint32_t`
em múltiplo de 4, `uint16_t` em múltiplo de 2). É decisão de performance:
CPUs leem mais rápido dados alinhados, e em algumas arquiteturas (ARM
antigo, SPARC) acessos desalinhados travam o programa.

Demonstração com a mesma struct nas duas versões:

```c
typedef struct {
  uint16_t a;   // 2 bytes
  uint32_t b;   // 4 bytes
  uint16_t c;   // 2 bytes
} Normal;

typedef struct {
  uint16_t a;
  uint32_t b;
  uint16_t c;
} __attribute__((packed)) Packed;
```

```
Normal: sizeof=12, offset(a)=0, offset(b)=4, offset(c)=8
Packed: sizeof=8,  offset(a)=0, offset(b)=2, offset(c)=6
```

Layout em memória:

```
Normal (12 bytes):              Packed (8 bytes):
+--+--+--+--+--+--+--+--+       +--+--+--+--+--+--+--+--+
| a  a  ?  ? | b  b  b  b |     | a  a  b  b  b  b  c  c |
| c  c  ?  ? |                  +--+--+--+--+--+--+--+--+
+--+--+--+--+
```

Os `?` são bytes de padding inseridos pelo compilador.

#### Por que isso quebraria a leitura do BMP

O cabeçalho BMP tem layout **exato no disco**: 14 bytes, com `bfSize`
começando no byte 2. Sem `packed`, `sizeof(BITMAPFILEHEADER)` viraria
**16** (compilador inseriria 2 bytes de padding entre `bfType` e
`bfSize` para alinhar o `DWORD`). Aí:

```c
fread(&bf, sizeof(BITMAPFILEHEADER), 1, inptr);
```

leria 16 bytes (2 a mais), e os campos seguintes viriam dos bytes
errados. O check em `filter.c:46` rejeitaria todo BMP válido.

`__attribute__((packed))` instrui o compilador: "empacote os campos sem
padding, mesmo que fiquem desalinhados". O custo é que acessar
`bf.bfSize` pode virar leitura desalinhada — em x86/ARM moderno funciona
(um pouco mais lento), em hardware mais restritivo pode falhar.

#### Regra prática

- **struct para representar dados na memória do programa**: deixe o
  compilador alinhar (sem `packed`). Performance grátis.
- **struct para mapear bytes de um arquivo, pacote de rede ou
  registrador de hardware**: use `__attribute__((packed))`. O layout é
  ditado pelo formato externo e o compilador não pode inventar bytes.

Detalhe de portabilidade: `__attribute__((packed))` é extensão
GCC/Clang.  C23 introduziu controle padronizado via `alignas`, mas
ainda não é universal.

### Projetos em C: separando `.h` e `.c`

O exemplo `src/filter/` é o primeiro do curso com **múltiplos arquivos
fonte**. A organização é:

- **`bmp.h`**: declaração de tipos (`BITMAPFILEHEADER`, `RGBTRIPLE`, etc).
  É um arquivo de **interface**.
- **`helpers.h`**: declaração de funções (`grayscale`, `reflect`, …) — só
  os protótipos. Outra interface.
- **`helpers.c`**: implementação dessas funções.
- **`filter.c`**: o programa específico que usa as funções de `helpers.h`.

A separação permite que `filter.c` apenas inclua `helpers.h` e use as
funções, sem precisar saber **como** elas são implementadas. Quem inclui
o `.h` precisa saber só dos protótipos; quem mantém `helpers.c` pode
mudar a implementação sem quebrar quem usa.

O `Makefile` orquestra a compilação:

```makefile
filter: filter.c helpers.c helpers.h bmp.h
    cc -o filter filter.c helpers.c
```

`cc` recebe os dois `.c` e linka num único executável. Os `.h` não são
"compilados" — são incluídos textualmente via `#include`.

### Debugando erros de memória — `slides-75.pdf`

Erros de gerência de memória em C dividem-se em algumas categorias
clássicas:

| problema        | causa típica                                            |
|-----------------|---------------------------------------------------------|
| memory leak     | `malloc` sem `free` correspondente                      |
| use-after-free  | acesso a ponteiro depois de `free` (o valor não é zerado) |
| buffer overflow | escrita além do tamanho alocado                         |
| heap overflow   | `malloc` de tamanho excessivo ou crescimento sem limite |
| stack overflow  | recursão sem caso base, frames muito grandes na pilha   |
| lixo na memória | leitura de variável local não inicializada              |

Em `slides-75.pdf` os exemplos `src/memory.c`, `src/garbage0.c` e
`src/garbage1.c` ilustram alguns desses casos.

Hoje rodamos apenas o exemplo do **memory leak no macOS**, para mostrar
como a ferramenta de detecção aponta a linha exata do `malloc` que ficou
sem `free`.

#### O exemplo: `src/memory.c`

```c
int *x = malloc(3 * sizeof(int));
x[0] = 72;
x[1] = 73;
x[2] = 33;
// programa termina sem free(x)
```

São 12 bytes pedidos no heap e nunca devolvidos. Ao final do processo o
sistema operacional recupera tudo, então um vazamento isolado num programa
curto não tem efeito visível. O problema aparece em processos de longa
duração (servidores, daemons): cada vazamento se acumula e a memória
disponível encolhe até o programa quebrar.

#### Detecção no macOS — `leaks`

A receita usada na aula:

```sh
clang -g memory.c -o memory
leaks --list --atExit -- ./memory
```

A flag `-g` pede ao compilador para embutir símbolos de depuração no
executável. Sem ela, `leaks` ainda detecta o vazamento, mas não consegue
apontar a linha do código fonte.

Para enriquecer o relatório com o stack trace completo do `malloc`
vazado, exporta-se `MallocStackLogging`:

```sh
export MallocStackLogging=1
leaks --list --atExit -- ./memory
unset MallocStackLogging
```

`leaks` então reporta algo como:

```
Process X: 1 leak for 12 total leaked bytes.
Leak: 0x... size=12 zone: ...
    Call stack: ... main + ... in memory.c:5
```

A linha `memory.c:5` é exatamente o `malloc` órfão.

#### Outras ferramentas (não rodadas hoje)

- **Valgrind** no Linux — detecta leaks, use-after-free, leitura de memória
  não inicializada, e mais.
- **AddressSanitizer (ASan)** do `clang`/`gcc` — instrumenta o programa em
  tempo de compilação (`-fsanitize=address`) e reporta os mesmos tipos de
  erro com penalidade de runtime menor que `valgrind`. Documentação em
  <https://clang.llvm.org>.

inicializado (lixo na pilha) e `garbage1.c` tem dois bugs combinados:
escrita em ponteiro `NULL` e *use-after-free*.

### `realloc` — `../semana-8/slides-81.pdf`

Já vimos que arrays no heap (`malloc`) têm tamanho fixo. E se durante a
execução descobrimos que precisamos de mais espaço?

A solução manual é **alocar novo, copiar, liberar antigo**:

```c
int *list = malloc(3 * sizeof(int));
list[0] = 1; list[1] = 2; list[2] = 3;

int *new = malloc(10 * sizeof(int));
for (int i = 0; i < 10; i++)
    new[i] = i < 3 ? list[i] : 0;

free(list);   // libera o array antigo
list = new;   // ponteiro passa a apontar para o novo bloco
```

A função `realloc` (da `<stdlib.h>`) encapsula esse padrão:

```c
list = realloc(list, 10 * sizeof(int));
```

Internamente, `realloc` decide entre duas estratégias:

- **Estender no mesmo lugar** se houver espaço livre adjacente — não copia
  bytes, apenas amplia o bloco. Caso ótimo.
- **Alocar novo bloco maior, copiar e liberar o antigo** se não houver
  espaço — análogo ao código manual acima.

O programador não precisa saber qual foi escolhida. `realloc` devolve um
ponteiro válido para o bloco redimensionado, ou `NULL` se a alocação
falhou (nesse caso o bloco original continua válido).

⚠️ As posições novas **não são zeradas**. Se cresci o array de 3 para 10
inteiros, os 7 últimos contêm lixo até serem inicializados.

### Listas encadeadas — `../semana-8/slides-82.pdf, slides-83.pdf`

Arrays exigem espaço **contíguo** na memória. Listas encadeadas dispensam
isso: cada nó vive num bloco próprio na heap, e cada bloco guarda o
endereço do próximo. Estrutura recursiva:

```c
typedef struct node {
    int number;
    struct node *next;
} t_node;
```

Cada nó (cons cell) carrega um valor e um ponteiro para o próximo. Lista
vazia = `NULL`. Lista de um elemento = um nó cujo `next == NULL`.

#### Comparação com array

| operação                          | array     | lista encadeada |
|-----------------------------------|-----------|-----------------|
| acessar elemento de índice $i$    | $O(1)$    | $O(n)$          |
| inserir no início                 | $O(n)$    | $O(1)$          |
| inserir no fim (com tail tracker) | $O(1)$    | $O(n)$          |
| inserir no meio                   | $O(n)$    | $O(n)$          |
| busca em estrutura não ordenada   | $O(n)$    | $O(n)$          |
| crescimento ilimitado             | realocação| natural         |

A escolha depende de quais operações dominam o problema.

#### Inserir no início — `list_insert`

```c
t_node* list_insert(int element, t_node* list) {
    t_node* n = malloc(sizeof(t_node));
    n->number = element;
    n->next = list;     // novo nó aponta para o antigo cabeça
    return n;            // retorna o novo cabeça
}

t_node *my = NULL;
my = list_insert(5, my);
my = list_insert(3, my);   // lista agora é 3 → 5 → NULL
```

Custo $O(1)$. Não importa o tamanho da lista.

#### Percorrer — `list_print`, `list_size`, `list_nth`

```c
void list_print(t_node *list) {
    for (t_node *t = list; t != NULL; t = t->next)
        printf("%i ", t->number);
    printf("\n");
}
```

O idioma `for (t = list; t != NULL; t = t->next)` é o equivalente em
listas encadeadas do `for (i = 0; i < n; i++)` em arrays. Custo $O(n)$.

`list_nth(list, n)` percorre até o $n$-ésimo nó — também $O(n)$, e ainda
**não é função total**: se `n` for maior que o tamanho da lista, o que
fazer? A implementação do slide chama `exit(1)`; uma alternativa mais
suave seria devolver código de erro via parâmetro de saída.

#### Reverter — três variações

A operação `reverse` é onde a aula gastou mais tempo, porque ela exibe
três estilos de implementação com trade-offs muito diferentes.

**(a) `list_reverse` — não destrutiva**

```c
t_node* list_reverse(t_node *head) {
    t_node *new = NULL;
    for (t_node *t = head; t != NULL; t = t->next)
        new = list_insert(t->number, new);
    return new;
}
```

Constrói uma **nova** lista invertida, sem modificar a original. Reusa
`list_insert`, que já insere no início — perfeito para inverter
naturalmente. Custo $O(n)$ em tempo, $O(n)$ em memória extra.

**(b) `list_nreverse1` — destrutiva, assinatura enganadora**

```c
t_node* list_nreverse1(t_node *head) {
    t_node *prev = NULL;
    t_node *current = head;
    while (current != NULL) {
        t_node *next = current->next;
        current->next = prev;   // reverte o ponteiro
        prev = current;
        current = next;
    }
    return prev;
}
```

Esta versão inverte **a própria lista** mutando os ponteiros `next`. Não
aloca memória nova. Mas a assinatura é traiçoeira:

```c
list_nreverse1(my);   // ⚠️ a mutação acontece, mas o resultado é descartado!
```

O cabeça original (`my`) passa a ser, depois da inversão, o **último** nó
da lista — agora aponta para `NULL`. O chamador continua segurando `my`,
que aponta para um único nó solto, com o resto da lista perdido (mas
ainda ligado via `prev`).

O uso correto é capturar o retorno:

```c
my = list_nreverse1(my);
```

A assinatura "parece com `list_reverse`" mas o comportamento é
fundamentalmente diferente. É **muito** propenso a erro.

**(c) `list_nreverse2` — destrutiva, com ponteiro para ponteiro**

```c
void list_nreverse2(t_node **head) {
    t_node *prev = NULL;
    t_node *current = *head;
    while (current != NULL) {
        t_node *next = current->next;
        current->next = prev;
        prev = current;
        current = next;
    }
    *head = prev;       // atualiza a referência do chamador
}
```

A diferença chave: recebe `t_node **head` e atualiza `*head`. Agora o
chamador escreve:

```c
list_nreverse2(&my);
// my já aponta para o novo cabeça automaticamente
```

A assinatura exibe explicitamente que **o ponteiro do chamador vai
mudar**. Mais difícil de usar errado. É o padrão idiomático em C
quando uma função precisa modificar um ponteiro do chamador.

#### Estruturas compartilhadas — `nconc` vs `append`

```c
t_node* list_nconc(t_node *a, t_node *b) {
    if (a == NULL) return b;
    t_node *t;
    for (t = a; t->next != NULL; t = t->next);
    t->next = b;       // anexa b ao fim de a, mutando o último nó de a
    return a;
}
```

`nconc` ("Non-CONsing Concatenate") **muta** o último nó de `a` para
apontar para o cabeça de `b`. Custo $O(n_a)$ (precisa achar o fim de
`a`). Após `list_nconc(a, b)`:

- `a` agora aponta para a lista combinada $a + b$
- `b` continua válido, mas seus nós são **compartilhados** com `a`

`list_append` é a versão não destrutiva:

```c
t_node* list_append(t_node *a, t_node *b) {
    t_node *new = list_copy(a);   // copia os nós de a
    list_nconc(new, b);
    return new;
}
```

Aqui `a` permanece intocada, mas os nós de `b` ainda são **compartilhados**
com a lista resultante. Trade-off:

- ✅ economiza memória (não copia `b`)
- ❌ se algo modificar a lista resultante alterando os nós originais de
  `b`, `b` muda junto. Bug sutil quando o chamador não espera.

A convenção `n`-prefix para versões destrutivas (`nreverse`, `nconc`) vem
do Common Lisp. No exemplo do slide, em CL:

```lisp
(let ((a (list 1 2 3)) (b (list 10 20 30)))
   (values a b (append a b)))
;=> (1 2 3) (10 20 30) (1 2 3 10 20 30)   ; a e b intocados

(let ((a (list 1 2 3)) (b (list 10 20 30)))
   (values a b (nconc a b)))
;=> (1 2 3 10 20 30) (10 20 30) (1 2 3 10 20 30)   ; a foi mutado!
```
