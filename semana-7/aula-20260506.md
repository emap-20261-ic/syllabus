## 1. Stack e heap

Até agora todas as variáveis que declaramos vivem na **stack** — uma região
de memória gerenciada automaticamente pelo compilador. Quando uma função é
chamada, suas variáveis locais são empilhadas; quando ela retorna, esse
espaço é liberado automaticamente.

A stack tem duas limitações importantes:

1. **O tamanho precisa ser conhecido em tempo de compilação.** Quando
   declaramos `int v[100]`, o `100` é constante. Em vários problemas até
   agora "alocávamos com folga" (um array maior do que o necessário) por
   não saber a quantidade exata.
2. **Variáveis morrem ao fim do escopo.** Não dá para retornar um ponteiro
   para uma variável local e usá-lo depois — a memória já foi reciclada.

Para os casos em que só sabemos o tamanho **em tempo de execução** (lendo
um arquivo, recebendo entrada do usuário, processando dados de tamanho
variável), usamos a outra região: a **heap**. A heap é gerenciada pelo
programador — você pede memória quando precisa e devolve quando termina.

## 2. `malloc`, `sizeof` e `free`

A função `malloc` (em `<stdlib.h>`) recebe o **número de bytes** que você
quer e retorna um ponteiro para o início do bloco recém-alocado. Se não
houver memória disponível, retorna `NULL`.

```c
int *px = malloc(sizeof(int));   // bloco de 4 bytes na heap
```

O operador `sizeof` retorna o tamanho em bytes do tipo (ou de uma
expressão). Usar `sizeof(int)` em vez de `4` torna o código portável: em
arquiteturas onde `int` tem tamanho diferente, o programa continua
correto.

Para alocar um array dinâmico:

```c
int x = get_info_from_user();
float *arr = malloc(x * sizeof(float));   // array de x floats na heap
```

Compare com `float arr[x]` (VLA — variable-length array): esse vive na
stack e desaparece ao fim da função. O alocado com `malloc` persiste até
você liberá-lo.

A liberação é explícita:

```c
free(px);
```

### Regras importantes

- Todo bloco alocado com `malloc` deve ser liberado com `free`.
- `free` só pode ser chamado em ponteiros que vieram de `malloc` (ou
  `calloc`/`realloc`).
- Não chame `free` no mesmo bloco duas vezes (*double free* — bug grave).

Memória alocada e nunca liberada é um **memory leak**. Em programas
longos, leaks degradam performance até o sistema ficar sem memória.

Veja slide-1.pdf

Os dois últimos pontos são essenciais:

- Depois de `free(b)`, **o ponteiro `b` ainda existe** na stack e ainda
  contém o endereço antigo. Mas o bloco para o qual ele aponta não é mais
  seu. Usá-lo é um bug (*use after free*).
- Como atribuímos `a = b` antes do `free`, **`a` também é um ponteiro
  inválido** depois do `free`. Múltiplas variáveis podem apontar para o
  mesmo bloco; basta uma chamar `free` para todas virarem inválidas.

## 4. Cópia de strings — por que `malloc` é necessário

Voltamos ao problema de fazer uma cópia editável de uma string. A
progressão `copy0.c` → `copy1.c` → `copy2.c` mostra três tentativas:

### 4.1 `copy0.c` — falsa cópia

```c
char s[] = "hi!";
char *t = s;
s[0] = toupper(s[0]);
printf("s: %s\nt: %s\n", s, t);
```

Saída:

```
s: Hi!
t: Hi!
```

**`t` não é uma cópia.** É só outro nome para o mesmo array `s` — modificar
um modifica o outro. Para ter uma cópia de verdade, precisamos de outro
bloco de memória.

### 4.2 `copy1.c` — alocando e copiando byte a byte

```c
char s[] = "hi!";
char *t = malloc(strlen(s) + 1);

for (int i = 0, n = strlen(s); i <= n; i++)
  t[i] = s[i];
t[0] = toupper(t[0]);
```

`malloc(strlen(s) + 1)` reserva `strlen` bytes para os caracteres mais
**1 byte para o terminador** `\0`. O laço copia tudo, inclusive o `\0`
(`i <= n`). Modificar `t` agora não afeta `s`.

### 4.3 `copy2.c` — usando `strcpy`

```c
const char *s = "hi!";
char *t = malloc(strlen(s) + 1);
strcpy(t, s);
t[0] = toupper(t[0]);
```

`strcpy` faz exatamente o que o laço fazia, mas em uma chamada. É a forma
idiomática em C para copiar strings.

`s` aponta para uma string literal (que pode estar em uma região de
read-only); `t` aponta para um bloco na heap que é nosso para
modificar e que devemos eventualmente liberar com `free(t)`.

## 5. `memory.c` — o exemplo mais simples de leak

```c
int main(void) {
  int *x = malloc(3 * sizeof(int));
  x[0] = 72;
  x[1] = 73;
  x[2] = 33;
}
```

O programa aloca 12 bytes e nunca os libera. Quando `main` retorna, o
sistema operacional recupera tudo de qualquer jeito, então neste caso o
"vazamento" é inofensivo. Mas em um programa que aloca dentro de um laço
ou de uma função chamada milhares de vezes, leaks como esse esgotam a
memória disponível. Vamos ver na próxima aula como detectá-los com
ferramentas como `leaks` (macOS) e `valgrind` (Linux).

A correção aqui seria simplesmente:

```c
free(x);
return 0;
```
