## Por que separar a biblioteca do programa

Até agora, cada programa em C que escrevemos era um único `.c`
compilado de uma vez. Mas as funções de lista encadeada vistas na
[aula 24](../semana-7/aula-20260508.md) (`list_insert`, `list_reverse`,
`list_copy`, …) são úteis em vários programas diferentes — não faz
sentido duplicá-las em cada arquivo que precise de listas.

A solução é tratar a coleção de funções como uma **biblioteca**: um
artefato compilado uma vez, em separado, e reaproveitado pelos
programas que dela precisarem. Essa separação tem duas vantagens
imediatas:

1. A biblioteca pode ser compilada de forma independente. Quando o
   código ficar **estável**, raramente precisará ser recompilada.
2. Quem usa a biblioteca não precisa enxergar a implementação — basta
   incluir o cabeçalho com os protótipos (assinaturas das funções).

## Interface vs. implementação: `list.h` e `list.c`

A divisão é a mesma já vista no exemplo `src/filter/` da
[aula 23](../semana-7/aula-20260508.md):

- **`list.h`** é a **interface**: declara o tipo opaco `t_node` e os
  protótipos das funções públicas. Quem incluir este arquivo sabe
  *o que* a biblioteca oferece, sem ver *como* ela implementa.

  ```c
  typedef struct node t_node;

  int     list_nth(t_node *list, int n);
  int     list_size(t_node *list);
  t_node* list_insert(int element, t_node* list);
  t_node* list_reverse(t_node *a);
  void    list_print(t_node *list);
  /* ... */
  ```

  A linha `typedef struct node t_node;` declara que existe um tipo
  `t_node` — mas **não diz seus campos**. Quem inclui `list.h` pode
  passar `t_node*` adiante, mas não pode acessar `node->next`
  diretamente. O tipo é "opaco": a estrutura interna fica encapsulada
  na biblioteca.

- **`list.c`** é a **implementação**: aqui o `struct node` é
  definido com seus campos `number` e `next`, e cada protótipo de
  `list.h` ganha corpo.

  ```c
  #include <stdio.h>
  #include <stdlib.h>
  #include "list.h"

  typedef struct node {
    int number;
    struct node *next;
  } t_node;

  /* implementações de list_insert, list_reverse, ... */
  ```

Note as aspas em `#include "list.h"` (em vez de `<list.h>`): aspas
indicam que o arquivo deve ser procurado primeiro no diretório do
fonte, e só depois nos diretórios de sistema. Bibliotecas locais usam
aspas; bibliotecas de sistema (como `<stdio.h>`) usam `<...>`.

## Compilação separada: `.o` e `.so`

Há duas formas principais de empacotar a biblioteca compilada,
correspondentes a duas filosofias de "linkagem" diferentes.

### Objeto: `list.o`

```sh
clang -c -Wall list.c
```

A flag `-c` instrui o compilador a **parar antes da linkagem**: ele
gera o arquivo objeto `list.o` (código de máquina, ainda sem
endereços resolvidos), mas não produz executável.

Para gerar o executável `test`, o arquivo objeto é então combinado
com o `.c` que usa a biblioteca:

```sh
clang -Wall list-test.c list.o -o test
```

O linker funde o `list.o` dentro do executável `test`. A partir desse
momento, **`test` é autossuficiente**: ele contém em si o código das
funções de lista, e pode ser distribuído sem `list.o`. Essa é a
**ligação estática**.

### Biblioteca compartilhada: `list.so`

```sh
clang -shared list.o -o list.so
```

`-shared` produz uma **shared library** (`.so` em
Linux/macOS, `.dll` em Windows). Compilando o programa contra ela:

```sh
clang -Wall list-test.c list.so -o test
```

Aqui o executável **não** carrega o código de `list.so`. Em vez
disso, ele guarda uma referência ao arquivo. Em tempo de execução, o
sistema operacional carrega `list.so` na memória e resolve os
símbolos. Essa é a **ligação dinâmica**.

### Demonstração: apagar o `.so`

Para evidenciar a diferença, durante a aula apagamos `list.so` depois
de compilar o `test`:

```sh
$ rm list.so
$ ./test
dyld: Library not loaded: list.so
  Referenced from: ./test
  Reason: image not found
[1]    abort      ./test
```

O executável quebra porque o `.so` que ele precisa carregar
em runtime sumiu. Já com a versão estática (gerada a partir do
`.o`), apagar `list.o` não afeta o executável — o código já está
embutido nele.

### Estática vs. dinâmica — trade-off

| característica              | estática (`.o`)        | dinâmica (`.so`)         |
|-----------------------------|------------------------|--------------------------|
| código no executável        | sim (copiado)          | não (referenciado)       |
| tamanho do executável       | maior                  | menor                    |
| precisa do `.so` em runtime | não                    | sim                      |
| atualizar a biblioteca      | requer recompilar tudo | basta substituir o `.so` |
| múltiplos programas         | cada um carrega cópia  | uma cópia compartilhada  |

Em `/usr/local/lib` e `/usr/lib` ficam dezenas de `.so` (ou `.dylib`
no macOS) usados por vários programas do sistema simultaneamente —
economiza memória e facilita atualizações de segurança.

## `Makefile`: automatizando a compilação

Compilar duas ou três linhas à mão funciona, mas se algum arquivo
mudar, é preciso lembrar quais comandos rodar de novo e em qual
ordem. O `make` resolve isso.

### Anatomia de uma regra

Cada bloco tem a forma:

```
alvo: dependências
	comando
```

A primeira linha lista o **alvo** (o que vai ser produzido) e suas
**dependências** (os arquivos de que o alvo depende). A linha indentada
(com **tab**, não espaços!) é o comando que produz o alvo.

A lógica do `make` quando processa o `src/list/Makefile`:

> "Para gerar `test`, preciso de `list.so`, `list.h` e `list-test.c`.
> Para gerar `list.so`, preciso de `list.o`. Para gerar `list.o`,
> preciso de `list.c`."

E o critério para decidir se cada comando precisa ser executado: o
`make` compara a **data de modificação** do alvo com a das
dependências. Se alguma dependência for mais recente que o alvo (ou
se o alvo não existir), o comando é executado. Caso contrário, é
pulado.

### Variáveis

```makefile
OPT=-Wall -Werror -Wpedantic -std=c17
COMPILER=clang
```

Variáveis são definidas com `=` e expandidas com `${nome}` (ou
`$(nome)`). Centralizam configuração: trocar de `clang` para `gcc` é
mudar uma linha.

### Alvos não-arquivo: `all` e `clean`

`all` e `clean` não correspondem a arquivos — são **alvos lógicos**.
`make all` dispara `test` (que dispara o resto), e `make clean`
apaga os artefatos gerados para forçar uma recompilação do zero.

### Por que a ordem importa

Sem `make`, a primeira compilação manual exige seguir a ordem certa:
`list.c` → `list.o` → `list.so` → `test`. O `make` deduz essa ordem
automaticamente a partir do grafo de dependências. Em projetos com
dezenas de arquivos, é a única forma sustentável.


## Implementação da conversão de Linked List para Array

Os arquivos do laboratório estão em [`semana-8/lab/`](lab/) e seguem o
mesmo padrão de `src/list/`: `list.h` (interface, não alterar),
`list.c` (implementação parcial, para completar), `test.c` (testes) e
`Makefile`. A descrição completa está em `lab/README.org`.

A primeira tarefa: implementar `list_array`, o **inverso** de
`list_create`. Recebe uma lista encadeada e copia seus elementos para
um array fornecido pelo chamador.

A assinatura em `list.h`:

```c
void list_array(t_node *list, int* ar, int size);
```

Pontos a observar:

- O array `ar` é fornecido pelo chamador (não alocado pela função).
  Quem chama precisa garantir que `ar` tem espaço para pelo menos
  `size` inteiros.
- O chamador tipicamente passa `list_size(list)` para o parâmetro
  `size`.
- O percurso da lista é o mesmo idioma de sempre:
  `for (t = list; t != NULL; t = t->next)`.

A implementação:

```c
void list_array(t_node *list, int ar[], int size) {
  int c = 0;
  for (t_node *tmp = list; tmp != NULL; tmp = tmp->next) {
    ar[c++] = tmp->number;
  }
}
```

O `test.c` já tem um caso de teste pronto. Cria um array, converte
para lista, converte de volta para array, e imprime. Saída esperada:

```
br[0] = 1
br[1] = 2
br[2] = 3
```

Para compilar e rodar:

```sh
$ make
$ ./test
```

O `Makefile` do laboratório segue o mesmo padrão de `src/list/`,
gerando `list.o`, `list.so` e finalmente o executável `test` que
linka contra o `.so`.
