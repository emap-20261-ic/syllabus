## Funções (slides-31)

### Boas práticas: `main` como orquestrador

Uma boa prática em C é manter `main` responsável apenas por entrada e saída
de dados, delegando o processamento para funções auxiliares. O arquivo
`semana-3/src/calculator.c` ilustra esse padrão:

```c
int main(void) {
    // entrada
    printf("x: "); scanf("%d", &x);
    printf("y: "); scanf("%d", &y);
    printf("operacao [a|s]: "); scanf(" %c", &op);

    // processamento (delegado)
    int status = calc(op, x, y, &result);

    // saída
    if (status == 0)
        printf("result = %d\n", result);
    else
        printf("Invalid Operation\n");
}
```

O código de negócio fica nas funções `add`, `subtract` e `calc`. A
função `main` apenas coordena.

### Protótipos de funções

Para que `main` possa chamar funções definidas *depois* dela no arquivo, é
necessário declarar os **protótipos** antes:

```c
int add(int a, int b);
int subtract(int a, int b);
int calc(char op, int a, int b, int* result);
```

O protótipo informa ao compilador a assinatura da função (tipos dos
parâmetros e do retorno) sem fornecer o corpo. A definição completa vem
mais adiante.

Como alternativa, também podemos declarar e inicializar (com a
implementação) de uma única vez antes da função `main`.

### `switch` como alternativa ao `if-else` aninhado

Quando precisamos testar um valor inteiro (ou `char`) contra vários casos
distintos, o `switch` é mais legível que uma cadeia de `if-else`:

```c
int calc(char op, int a, int b, int* r) {
    switch (op) {
    case 'a': case 'A':
        *r = add(a, b);
        break;
    case 's': case 'S':
        *r = subtract(a, b);
        break;
    default:
        return 1;
    }
    return 0;
}
```

Pontos a destacar:

- Dois `case` consecutivos sem `break` entre eles compartilham o mesmo
  corpo — aqui `'a'` e `'A'` executam o mesmo código.
- O `break` ao final de cada caso evita o **fall-through** (execução
  involuntária do próximo caso).
- O `default` é executado se nenhum `case` corresponder — equivale ao
  `else` final de um `if-else` aninhado.
- Sem o `break`, a execução "cai" para o próximo `case`: esse
  comportamento pode ser intencional (como nos casos `'a'`/`'A'`), mas
  costuma ser um bug quando não planejado.

### Overflow de `int`

O tipo `int` tem um intervalo limitado (tipicamente −2 147 483 648 a
2 147 483 647 em sistemas de 32 bits). Quando o resultado de uma operação
ultrapassa esse intervalo ocorre **overflow**: o valor "dá a volta" e
produz um resultado incorreto sem qualquer aviso em tempo de execução.

Demonstramos isso em `semana-3/src/calculator.c` somando dois valores
grandes — por exemplo, `2 000 000 000 + 2 000 000 000` retorna um número
negativo. A solução é usar um tipo de maior capacidade:

```c
long add(long a, long b) {
    return a + b;
}
```

`long` garante pelo menos 32 bits (tipicamente 64 em sistemas modernos);
`long long` garante pelo menos 64 bits. O especificador de formato
correspondente é `%ld` para `long` e `%lld` para `long long`.

## Ponteiros (slides-31)

### Motivação: retornar resultado *e* sinalizar erro

A função `calc` precisa:

1. calcular e devolver o resultado da operação; e
2. indicar se a operação é válida ou não.

Uma função em C tem **um único valor de retorno**. A solução é usar o
retorno para o código de status (0 = sucesso, 1 = erro) e passar o
resultado por meio de um **ponteiro**:

```c
int calc(char op, int a, int b, int* r) {
    // *r recebe o resultado; retorno indica sucesso/erro
}
```

Em `main`, chamamos passando o endereço de `result`:

```c
int result = 0;
int status = calc(op, x, y, &result);
if (status == 0) {
    printf("result = %d\n", result);
    return EXIT_SUCCESS;
} else {
    printf("Invalid Operation\n");
    return EXIT_FAILURE;
}
```

As constantes `EXIT_SUCCESS` (0) e `EXIT_FAILURE` (1) vêm de `<stdlib.h>`
e são a forma idiomática de retornar códigos de status do `main`. Usar
essas constantes em vez dos literais `0` e `1` torna a intenção do código
mais clara.

### Passagem por valor vs. por referência: `swap0.c` e `swap1.c`

**swap0.c** — não funciona:

```c
void swap(int a, int b) {
    int tmp = a;
    a = b;
    b = tmp;
}
```

`a` e `b` são **cópias** dos valores de `x` e `y`. A troca ocorre dentro
da função, mas `x` e `y` em `main` permanecem inalterados. C passa
argumentos **por valor**: a função recebe uma cópia, não o original.

**swap1.c** — funciona:

```c
void swap(int* a, int* b) {
    int tmp = *a;
    *a = *b;
    *b = tmp;
}
```

Passamos os **endereços** de `x` e `y`:

```c
swap(&x, &y);
```

Agora `a` e `b` são ponteiros — eles armazenam endereços de memória. O
operador `*` (desreferência) acessa ou modifica o valor *naquele endereço*.
A troca acontece diretamente nas variáveis originais.

### Operadores `&` e `*`

| Operador | Nome          | Uso                                        |
|----------|---------------|--------------------------------------------|
| `&x`     | endereço de   | obtém o endereço de memória de `x`         |
| `*p`     | desreferência | acessa/modifica o valor apontado por `p`   |

```c
int x = 10;
int* p = &x;   // p aponta para x
*p = 20;       // equivale a x = 20
```

O tipo `int*` se lê "ponteiro para int": uma variável que armazena o
endereço de um `int`.

## Depuração com `printf` (slides-31)

### O problema do `scanf(" %c", ...)`

Ao testar `calculator1.c`, a leitura do operador com `scanf("%c", &op)`
capturava o caractere `'\n'` deixado no buffer pelo `scanf` anterior
(que leu `y`), em vez de aguardar o caractere digitado pelo usuário.

Inserimos `printf` para inspecionar o valor de `op` e identificar o
problema. Com a ajuda do ChatGPT, descobrimos a solução: adicionar um
**espaço antes de `%c`** na string de formato:

```c
scanf(" %c", &op);
```

O espaço na string de formato instrui o `scanf` a **ignorar espaços em
branco** (incluindo `'\n'`) antes de tentar ler o caractere. Sem o espaço,
`%c` lê o próximo caractere disponível no buffer, que pode ser o `'\n'` da
leitura anterior.

### Depuração no VS Code

Mencionamos brevemente o depurador integrado do VS Code. A configuração é
mais trabalhosa do que simplesmente compilar e rodar, especialmente quando
o programa lê dados do terminal via `scanf` — o terminal integrado do
depurador pode se comportar de forma diferente do terminal comum.

## Arrays de números (slides-31)

Usando `semana-2/src/scores0.c` como exemplo, introduzimos arrays:

```c
int score1 = 72;
int score2 = 73;
int score3 = 33;
```

Três variáveis separadas é trabalhoso e não escala. Um **array** agrupa
elementos do mesmo tipo em sequência na memória:

```c
int scores[3] = {72, 73, 33};
```

- `scores[0]`, `scores[1]`, `scores[2]` acessam os elementos
  (índice começa em 0).
- O tamanho do array é fixo em tempo de compilação.
- Arrays em C não guardam seu próprio tamanho — o programador deve
  controlá-lo.

## Strings e arrays de caracteres (slides-32)

Uma **string** em C é um array de `char` terminado pelo caractere nulo
`'\0'`:

```c
char nome[6] = {'A', 'l', 'i', 'c', 'e', '\0'};
// ou, equivalentemente:
char nome[] = "Alice";
```

O compilador adiciona `'\0'` automaticamente ao final de um literal de
string. Por isso `"Alice"` ocupa 6 bytes, não 5.

Ponto importante: strings em C são mutáveis (diferente de algumas outras
linguagens), mas escrever além do tamanho alocado causa comportamento
indefinido.
