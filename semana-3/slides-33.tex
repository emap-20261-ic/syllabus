  Os \ilcode{char} são armazenados como inteiros (códigos ASCII).
  
  \framebreak

  \ilcode{printf} fez uma conversão \emph{implícita}, mas também
  podemos converter explicitamente. \vspace{.5cm}

  \begin{lstlisting}[style=normalc]
    printf("%i %i %i\n", (int) c1, (int) c2, (int) c3);
  \end{lstlisting}

  Mas conversões podem \emph{perder} informação, por exemplo, de
  \ilcode{float} para \ilcode{int}.
  
\end{frame}

\begin{frame}[fragile,allowframebreaks]{Strings}

  Representamos \emph{strings} como arrays de caracteres (cada um com
  8 bits ou um byte), um tipo que na verdade não é primitivo em C.

  \lstinputlisting[style=normalc,caption=src/hi2.c]{src/hi2.c}

  \framebreak

  \lstinputlisting[style=normalc,caption=src/hi3.c]{src/hi3.c}
  \begin{lstlisting}[style=normalbash]
    > ./a.out
    72 73 33
  \end{lstlisting}

  \framebreak

  \includegraphics[width=.45\textwidth]{chars.png}\hfill
  \includegraphics[width=.45\textwidth]{string.png}

  Em C, strings terminam com o char \ilcode{'\\0'} ou inteiro
  \ilcode{0}, ou um byte com todos os bits 0, marcando a
  \emph{terminação} da string. Este é o chamado 'NUL'. Então
  precisamos sempre de um byte adicional.

  Outros tipos de dados de tamanho \emph{fixo} não precisam de tal
  mecanismo. Mas bibliotecas que oferecem representação de variadas
  precisões numéricas podem usar também tal recurso.

  \framebreak

  \lstinputlisting[style=normalc,caption=src/hi4.c]{src/hi4.c}
  \begin{lstlisting}[style=normalbash]
    > ./a.out
    72 73 33 0
  \end{lstlisting}
