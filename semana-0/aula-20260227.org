* Revisão: Git e GitHub

** Conceitos revisados

Revisamos os conceitos fundamentais de Git e GitHub discutidos na aula
anterior:

- *Repositório* :: um diretório de arquivos rastreado pelo Git, com
  histórico completo de alterações.

- *Commit* :: um snapshot do estado dos arquivos em um determinado
  momento, acompanhado de uma mensagem descritiva.

- *Staging area* :: área intermediária onde marcamos quais alterações
  serão incluídas no próximo commit (usando =git add=).

- *Remote* :: o repositório remoto, hospedado no GitHub, que serve
  como ponto de sincronização entre diferentes máquinas ou
  colaboradores.

- *Push e Pull* :: =git push= envia commits locais para o remoto;
  =git pull= traz alterações do remoto para a máquina local.

** Demonstração com dois terminais

Para ilustrar o fluxo de trabalho com Git de forma mais concreta,
fizemos uma demonstração usando duas janelas de terminal abertas
simultaneamente no meu computador, simulando dois colaboradores (ou a
mesma pessoa trabalhando em duas máquinas diferentes).

O fluxo demonstrado foi:

1. No *terminal A*, criamos um repositório, adicionamos um arquivo e
   fizemos o push para o GitHub:

   #+begin_src bash
     $ cd repositorio
     $ git init 
     $ echo "conteudo inicial" > arquivo.txt
     $ git add arquivo.txt
     $ git commit -m "adiciona arquivo inicial"
     $ git push
   #+end_src

2. No *terminal B*, clonamos o mesmo repositório e verificamos que o
   arquivo estava lá:

   #+begin_src bash
   $ git clone <caminho-do-repositorio>
   $ cd repositorio
   $ cat arquivo.txt
   #+end_src

3. De volta ao *terminal A*, fizemos uma modificação e push:

   #+begin_src bash
   $ echo "nova linha" >> arquivo.txt
   $ git add arquivo.txt
   $ git commit -m "adiciona nova linha"
   $ git push
   #+end_src

4. No *terminal B*, usamos =git pull= para trazer a atualização:

   #+begin_src bash
   $ git pull
   $ cat arquivo.txt
   #+end_src

Esta demonstração tornou visível como o Git permite que diferentes
instâncias de trabalho se mantenham sincronizadas através do
repositório remoto no GitHub.

* GitHub Codespaces

Apresentamos o [[https://github.com/features/codespaces?locale=pt-BR][GitHub Codespaces]] como uma alternativa prática para
quem não quer ou não pode configurar um ambiente de desenvolvimento
local. O Codespaces oferece um ambiente de desenvolvimento completo
diretamente no navegador, baseado no VS Code.

Para demonstrar, usamos o repositório de uma aluna da turma. Os passos
para criar um Codespace são:

1. Acesse o repositório no GitHub.
2. Clique no botão verde *Code*.
3. Selecione a aba *Codespaces*.
4. Clique em *Create codespace on main*.

Em poucos segundos, o navegador abre uma interface do VS Code com o
repositório já clonado, um terminal integrado disponível e todas as
ferramentas necessárias para programar (incluindo o compilador =gcc=).

O Codespaces é especialmente útil porque:

- não requer instalação de nada no computador do aluno
- funciona em qualquer sistema operacional com um navegador
- o ambiente já vem configurado com as ferramentas que precisamos
- cada aluno tem uma cota gratuita mensal no plano educacional do
  GitHub

Por outro lado, o Codespaces depende de uma boa conexão com a
internet e pode haver períodos em que a performance é inferior ao que
seria em um ambiente local.

* WSL para usuários Windows

Para quem prefere trabalhar localmente no Windows, a alternativa
recomendada é o *WSL* (Windows Subsystem for Linux). O WSL permite
rodar um sistema Linux completo dentro do Windows, sem precisar de
máquina virtual tradicional ou dual boot.

A instalação é simples. No PowerShell (como administrador):

#+begin_src bash
wsl --install
#+end_src

Este comando instala por padrão a distribuição Ubuntu. Após a
instalação e reinicialização do computador, você terá acesso a um
terminal Linux com todos os comandos e ferramentas que usamos no
curso (=gcc=, =make=, =git=, etc.).

A documentação oficial está disponível em
https://learn.microsoft.com/en-us/windows/wsl/install.

Após instalar o WSL, caso você use o VS Code, pode ser necessária uma
configuração adicional para que o editor utilize o WSL e as
ferramentas nele instaladas, em especial o compilador =gcc=. Para
isso, instale a extensão [[https://marketplace.visualstudio.com/items?itemName=ms-vscode-remote.remote-wsl][WSL]] no VS Code. Com ela, o editor consegue
abrir projetos diretamente dentro do ambiente Linux do WSL, usando o
terminal, o compilador e as demais ferramentas instaladas lá.
